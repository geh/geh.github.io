¿Cuál es el punto?

Arresgiar incorrectitud para ahorrar tiempo.

Un algoritmo rapido con probabilidad de error muy baja puede ser más deseable que un algoritmo lento totalmente exacto.

En el mundo real, los cálculos tienen una probabilidad de no terminar, por más que en teoría sean determinísticos y terminan:

  • una computadora puede fallar por sus componientes internos
  • un cálculo puede fallar por causas externas (¿ideas?)

Incorrectitud como recurso

Usar incorrectitud puede querer decir:

  • para problemas de funciónes, buscar entre soluciones aproximadas de la solución correcta
  • para problemas de decisión, poder equivocarse con una probabilidad baja entre las corridas posibles

En el segundo caso, necesitamos aleatoriedad, ie, algoritmos que puedan aceder a bits aleatorios (tirar una moneda para obtener cara o ceca).

Introducir el azar en un algoritmo

Ejemplo: chequear bases de datos

Dos computadoras R1 R2 están separadas por una red. Queremos chequear que alojan los mismos datos. Representamos los datos como secuencias:

x1x2...xn
y1y2...yn

La tarea consiste en usar un canal de comunicación entre R1 y R2 para chequear que x = y.

Complejidad de la comunicación: cuantos bits hay que intercambiar.

¡Hay que mandar todo!

Decir que tenemos dos números muy grandes:

X = x1x2...xn
Y = y1y2...yn

Eligir número primo p al azar, chequear:


Xmodp = Ymodp

Si X = Y, entonces sus restos son iguales bajo cualquiera división. Si los dos números no son iguales, y si eligimos un número primo al azar, cuales son las chances que el resultado va a ser distinto?

La pregunta es si X − Y = 0modp, ie si p divide X − Y.

Cuantos números dividen X − Y? Sea n el número de bits para escribir X − Y. Hay como máximo n divididores primos.

Según el teorema de los numeros primos, hay más de M/(logM) numero primos hasta M.

Eligiendo un primo con 10log(n) bits (ie p vale hasta n10), el número de primos hasta n10 es bastante grande (n10/10log(n)). Pero el número de divisores de X − Y es como máximo n.

Entonces eligiendo un primo al azar tenemos muchas chances de encontrar uno que no divide X − Y.

Entonces comparar dos databases se puede hacer en O(log(n)) bits intercambiados con alta probabilidad.

(Para n = 1016, mandar 256 bits.)

Se puede bajar la probabilidad de error tirando al azar más primos, y aumentando la complejidad de la comunicación sólo por un factor constante!

Ej: 10 primos

(2(logn)/n)10 = 210 ⋅ (logn)10/n10

si n = 1016, error(x, y) ≤ 0.48/10141

(hay menos de 1099 protones en el universo)

Igualdad de polinomios

Tenemos dos polinomios de grado d, descritos de manera compacta, queremos chequear si son iguales o no:


(1 − x)5(5x4 + 7x)3 − (x − 5)64 = (x − 4)8(x2 + 5x)425?

Desarrollar los términos demora un tiempo exponencial en función del tamaño inicial de los polinomios.

Pero eligir un x aleatorio, y metiéndolo en los polinomios para comparar sus valores en x es mucho más rápido.

Si los dos polinomios son distintos, en cuantas valores pueden estar iguales? En d valores, según el teorema fundamental del algébra. Y hay poca chance que un real aleatorio les haga coincidir.

P versus BPP

Probabilidades

  • A un evento
  • Pr[A] probabilidad que A pase
  • Pr[¬A] = 1 − Pr[A]
  • Pr[A ∨ B] = Pr[A] + Pr[B] − Pr[A ∧ B] ≤ Pr[A] + Pr[B] (desigualdad de Boole)
  • Pr[A ∧ B] = Pr[A]Pr[B] ssi A y B son independientes
  • Pr[A ∣ B] = Pr[A ∧ B]/Pr[B] = Pr[B ∣ A]Pr[A]/Pr[B] regla de Bayes
  • x variable random
  • Ex[X] = ΣiPr[X = i]i esperanza de X: valor medio de un fenómeno aleatorio
  • Ex[X + Y] = Ex[X] + Ex[Y] linearidad de la esperanza
  • Ex[XY] = Ex[X]Ex[Y] solamente si X y Y son independientes

Desigualdad de Markov

Si la distribucion X toma valores positivos:

Pr[X ≥ k ⋅ E[X]] ≤ 1/k

Máquina de Turing Probabilística

Una MTP es una MT con dos funciones de transición δ0 y δ1. Para ejecutar una MTP M con un input x, se elige en casa paso con probabilidad 1/2 entre aplicar δ0 y δ1.

La máquina sale 1 (aceptar) o 0 (rechazar). Denotamos M(x) la variable aleatoria correspondiente al valor que M escribe al final de su cálculo. Dada una función T : N ↦ N, decimos que M corre en tiempo T(n) si para todo input x, M se detiene con input x en T(∣x∣) pasos independientemente de sus elecciones aleatorias.

Pr[M(x) = 1] es la fracción de ramas de computación de M que se terminan con output 1.

Diferencias MTP vs MTND

La diferencia con MTND es la interpretación del grafo de las computaciones posibles. Una MTND acepta x si existe 1 rama exitosa.

Una MTP representa una computación realística.

BPP

Dado T : N ↦ N y L ⊆ {0, 1} *  decimos que una MTP M decide L en tiempo T(n) si para todo x ∈ {0, 1} * , M se detiene en T(∣x∣) pasos independientemente de sus elecciones aleatorias, y Pr[M(x) = L(x)] ≥ 2/3.

Sea BPTIME(T(n)) la clase de lenguajes decididos por MTP en tiempo O(T(n)) y definimos:

BPP =⋃c BPTIME(nc)

Observaciones

  • para un input fijado x, M sale el output correcto L(x) con probabilidad  ≥ 2/3
  • una TM determinística es un caso particular de MTP entonces P ⊆ BPP
  • la constante 2/3 es arbitraria entre las constantes  > 1/2, porque se puede amplificar

¿Cuantas veces para amplificar?

¿Como aumentar la correctitud de probabilidad p a q?

Desigualdad de Chernoff, para toda constante a, existe ca < 1 constante tal que:

Pr[∣X − n/2∣ > a ⋅ n] ≥ can

Ie repitiendo un algoritmo BPP n veces, se decrece la probabilidad que la mayoría de las respuestas sean falsas de manera exponencial en n.

Demo

Sea c constante.

E[cX] = E[cX1 + X2 + … + Xn]
 = E[cX1 ⋅ cX2 ⋅ … ⋅ cXN]
 = E[cX1] ⋅ E[cX2] ⋅ … ⋅ E[cXN]
 = ((1 + c)/2)n

Por la desigualdad de Markov:

Pr[cX ≥ k((1 + c)/2)n] ≤ 1/k

Pr[X ≥ logc(k) + n ⋅ logc((1 + c)/2)] ≤ 1/k

Falta eligir una constante c > 1 adecuada para encontrar la desigualdad de Chernoff.

Haciendo más grande d = logck (la deviación de X del promedio), la probabilidad que X devie de esa cantidad baja exponencialmente en función de d.

Definición alternativa

L ∈ BPP si existe una MT corriendo en tiempo polinomial y un polinomio p : N ↦ N tal que para todo x ∈ {0, 1} * :

Prr ∈ {0, 1}p(∣x∣)[M(x, r) = L(x)] ≥ 2/3

Obs.

  • BPP ⊆ EXP dado que en tiempo 2nk se puede enumerar todas las secuencias de una MTP polinomial y correr esa MT según cada una de ellas.
  • P ⊆ BPP ⊆ EXP es lo que sabemos hoy
  • se sospecha que BPP = P (por derandomización)

Otras clases

BPP tiene errores de los dos lados. Un algoritmo para comparar números o polinomios como nuestros ejemplos del principio, hace errores solo en un lado.

Clases RP y coRP.

RP

L ∈  RTIME(T(n)) si existe una MTP M corriendo en tiempo T(n) tal que:

  • x ∈ L implica Pr[M(x) = 1] ≥ 2/3
  • $x\notin L$ implica Pr[M(x) = 0] = 1

Definimos RP = ⋃c RTIME(nc)

  • si M(x) = 1 entonces x ∈ L
  • si M(x) = 0 entonces x ∈ L o $x\notin L$, no sabemos

coRP es el complemento de RP.

Zero-sided error

Ejemplo: encontrar la mediana

Una mediana para un conjunto de números {a1, …, an} es un nûmero x tal que por lo menos ⌊n/2⌋ de los ai son más chicos o iguales a x y por lo menos ⌊n/2⌋ de ellos son más grandes o iguales a x.

Un método para encontrar la mediana es ordenar la lista y salir el ⌊n/2⌋ésimo número más chiquito, pero eso lleva un tiempo O(n ⋅ log(n)).

Existe un algoritmo probabilístico que encuentra la mediana en tiempo O(n). (También existen determinísticos corriendo en tiempo lineal pero son más complicados.)

Presentamos un alg. que resuelve un caso mǽs general: encontrar el késimo número más chiquito de un conjunto para k dado.

FindKthElement(k, a1, …, an):

  1. Eligir un x = ai al azar
  2. Contar m el número de ai de la lista tales que ai ≤ x
  3. If m = k, entonces salir k
  4. Sino, si m > k, copiar en una lista nueva L los elementos ai ≤ x y correr FindKthElement(k, L)
  5. Sino (si m < k), copiar en una lista nueva H los elementos ai > x y correr FindKthElement(k − m, L)
